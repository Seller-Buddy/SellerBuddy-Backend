# CS 에이전트 소개

## 1. CS 에이전트 한 줄 소개

운영자가 고객 문의와 주문 정보를 입력하면 AI가 문의 유형을 분류하고, 판단에 필요한 정보가 빠졌는지 확인한 뒤, ChromaDB에 저장된 쇼핑몰 정책을 RAG로 검색합니다.

검색한 정책과 고객 상황을 비교해 처리 가능성을 판단하고, 정책 근거가 포함된 고객 답변 초안을 작성합니다. 마지막으로 확답 위험과 예외 상황을 검수하고 운영자 승인 필요 여부까지 반환하는 FastAPI 기반 CS 에이전트 워크플로우입니다.

단순히 LLM에게 "고객 답변을 작성해줘"라고 요청하는 구조가 아니라,

```text
정책 등록
→ 문의 분류
→ 필수 정보 확인
→ 정책 검색
→ 처리 가능성 판단
→ 답변 초안 작성
→ 안전 검수
→ 운영자 승인 확인
```

순서로 동작하는 다단계 파이프라인입니다.

---

## 2. 전체 서비스 흐름

CS 기능은 정책을 ChromaDB에 등록하는 흐름과 고객 문의를 분석하는 흐름으로 나뉩니다.

### 정책 등록 흐름

```text
운영자의 정책 JSON 입력
↓
정책 제목·본문·카테고리·출처 확인
↓
정책 본문 청크 분할
↓
Passage Embedding 생성
↓
정책 메타데이터 구성
↓
ChromaDB에 저장
```

### 고객 문의 분석 흐름

```text
운영자의 고객 문의 입력
↓
문의 접수/분류 Agent
↓
정보 확인 Agent
↓
정책 검색 Agent
↓
판단 Agent
↓
답변 작성 Agent
↓
안전 검수 Agent
↓
운영자 승인 Agent
↓
분석 결과와 고객 답변 초안 반환
```

고객 문의는 검색 시점에 Query Embedding으로 변환되지만 ChromaDB에 저장되지는 않습니다. ChromaDB에는 운영자가 등록한 정책 문서만 저장됩니다.

## 3. 에이전트 구조

이 프로젝트의 CS 백엔드는 여러 에이전트가 역할을 나누어 순서대로 동작하도록 설계되어 있습니다.

현재 CS 워크플로우는 일곱 단계로 구성됩니다.

```text
inquiry_classifier
→ 고객 문의를 요약하고 문의 유형을 분류

missing_info_checker
→ 판단에 필요한 정보가 빠졌는지 확인

policy_search
→ 고객 문의와 관련된 쇼핑몰 정책을 ChromaDB에서 검색

decision
→ 고객 상황과 정책 조건을 비교해 처리 가능성을 판단

response_writer
→ 정책 근거가 포함된 고객 답변 초안을 작성

safety_review
→ 확답 위험, 정책 부족, 분쟁 가능성, 예외 상황을 검수

operator_approval
→ 운영자가 반드시 확인해야 하는 문의인지 결정
```

`inquiry_classifier`는 명확한 업무 키워드를 먼저 적용하고, 규칙만으로 유형을 정하기 어려운 문의에 한해 LLM을 사용합니다.

`missing_info_checker`, `decision`, `response_writer`, `safety_review`, `operator_approval`은 정책과 명시적인 업무 규칙을 중심으로 동작합니다. `policy_search`는 Upstage Embedding API와 ChromaDB를 이용합니다.

## 4. 정책 등록 및 검색 흐름

정책 검색 Agent가 동작하려면 먼저 쇼핑몰 정책을 ChromaDB에 등록해야 합니다.

정책 데이터는 다음 형식으로 전달됩니다.

```json
{
  "documents": [
    {
      "title": "교환 및 반품 정책",
      "content": "상품 수령 후 7일 이내이며 미사용이고 택이 유지된 경우 교환할 수 있습니다.",
      "category": "exchange",
      "source": "쇼핑몰 이용약관"
    }
  ],
  "reset_collection": false
}
```

### 정책 등록 전체 흐름

```text
정책 documents 입력
↓
빈 정책 여부 확인
↓
정책 본문을 문단 단위로 분할
↓
긴 문단을 더 작은 청크로 분할
↓
각 청크의 고유 ID 생성
↓
Passage Embedding 생성
↓
본문·메타데이터·임베딩을 ChromaDB에 upsert
↓
저장 문서 수와 청크 수 반환
```

### 정책 검색 전체 흐름

```text
문의 유형 + 고객 문의 + 주문 정보 입력
↓
검색용 문장 구성
↓
Query Embedding 생성
↓
ChromaDB 코사인 유사도 검색
↓
관련 정책 상위 결과 조회
↓
정책 제목·발췌문·점수·카테고리·출처 반환
```

## 5. 정책 등록 및 검색 내부 함수/로직 설명

### 5-1. 정책 입력 및 검증 로직

정책 등록 API는 `documents` 배열을 입력받습니다. 각 정책은 다음 필드를 가집니다.

```text
title
→ 정책 제목

content
→ 임베딩하고 검색할 정책 본문

category
→ 환불, 교환, 배송 등 정책 유형

source
→ 정책의 출처 또는 문서명
```

정책 본문을 분할한 결과가 비어 있으면 저장을 중단하고 입력 오류를 반환합니다.

#### 발표에서 설명할 포인트

정책을 단순 문자열로만 저장하지 않고 제목, 유형, 출처를 함께 관리합니다. 검색 결과에 정책의 근거 정보를 포함하기 위한 구조입니다.

### 5-2. 정책 청크 분할 로직

정책 전체를 하나의 긴 텍스트로 임베딩하지 않고 빈 줄을 기준으로 문단을 분리합니다.

```text
정책 content 입력
↓
연속된 빈 줄을 기준으로 문단 분리
↓
공백과 줄바꿈 정리
↓
900자 이하 문단은 하나의 청크로 사용
↓
900자를 초과하는 문단은 800자 단위로 분할
```

각 청크에는 정책 제목, 카테고리, 출처, 청크 번호가 메타데이터로 연결됩니다.

#### 발표에서 설명할 포인트

정책 전체를 하나의 벡터로 만들면 환불, 교환, 배송처럼 서로 다른 의미가 섞일 수 있습니다. 정책을 작은 의미 단위로 나누어 고객 문의와 직접 관련된 문장을 더 정확히 검색하도록 했습니다.

### 5-3. 정책 청크 ID 생성 로직

정책 청크의 ID는 다음 정보를 조합한 뒤 SHA-1 해시를 생성해 만듭니다.

```text
정책 제목
+ 정책 카테고리
+ 정책 출처
+ 청크 번호
+ 청크 본문
```

생성된 ID는 `policy_해시값` 형식으로 사용됩니다. ChromaDB에는 `upsert` 방식으로 저장하므로 같은 ID가 다시 들어오면 새 항목을 무조건 추가하지 않고 기존 항목을 갱신합니다.

#### 발표에서 설명할 포인트

정책의 위치와 본문을 기반으로 안정적인 ID를 생성합니다. 동일한 정책을 다시 등록할 때 불필요한 중복 데이터가 만들어지는 것을 줄이는 장치입니다.

### 5-4. Passage Embedding 및 ChromaDB 저장 로직

정책 청크는 문서 검색에 특화된 모델로 임베딩합니다.

```text
모델: solar-embedding-1-large-passage
저장소: ChromaDB PersistentClient
기본 컬렉션: shopbuddy_policies
거리 기준: cosine
```

ChromaDB에는 다음 데이터가 함께 저장됩니다.

```text
ids
→ 정책 청크의 고유 ID

documents
→ 실제 정책 청크 본문

metadatas
→ title, category, source, chunk_index

embeddings
→ Passage Embedding 벡터
```

`reset_collection`이 `true`이면 기존 정책 컬렉션을 삭제한 뒤 새 컬렉션을 생성합니다. `false`이면 기존 컬렉션을 유지한 상태에서 정책을 추가하거나 갱신합니다.

#### 발표에서 설명할 포인트

정책은 문의가 들어올 때마다 다시 임베딩하지 않습니다. 한 번 임베딩한 정책을 ChromaDB에 영구 저장하고 이후 검색에서 재사용합니다.

### 5-5. Query Embedding 검색 로직

고객 문의를 검색할 때는 질문과 검색에 특화된 Query Embedding 모델을 사용합니다.

```text
문의 유형: exchange
고객 문의: 어제 받은 상품의 사이즈를 교환하고 싶어요.
주문/상품 정보: 미사용, 택 유지
```

위 정보를 하나의 검색 문장으로 구성한 뒤 `solar-embedding-1-large-query` 모델로 임베딩합니다.

생성된 문의 벡터는 ChromaDB에 저장하지 않고 정책 벡터와 비교하는 검색 입력으로만 사용됩니다.

기본적으로 관련 정책 상위 3개를 검색하며, 요청의 `top_k` 값에 따라 1개에서 최대 10개까지 조회할 수 있습니다.

#### 발표에서 설명할 포인트

검색할 문서는 Passage Embedding, 검색 질문은 Query Embedding을 사용합니다. 저장 대상과 검색 대상의 역할에 맞게 서로 다른 임베딩 모델을 적용했습니다.

### 5-6. 검색 결과 정규화 로직

ChromaDB의 검색 결과를 그대로 뒤 단계에 전달하지 않고 다음 구조로 정리합니다.

```json
{
  "title": "교환 및 반품 정책",
  "excerpt": "상품 수령 후 7일 이내이며 미사용이고 택이 유지된 경우...",
  "score": 0.87,
  "category": "exchange",
  "source": "쇼핑몰 이용약관"
}
```

코사인 거리는 `1 - distance` 방식으로 점수화하고 0 미만의 값은 0으로 처리합니다. 긴 정책 본문은 답변 프롬프트가 지나치게 커지지 않도록 발췌문 길이를 제한합니다.

#### 발표에서 설명할 포인트

검색 결과를 판단 Agent와 답변 작성 Agent가 바로 사용할 수 있는 표준 구조로 변환합니다. 정책 내용뿐 아니라 제목, 출처, 검색 점수를 함께 제공한다는 점이 중요합니다.

## 6. CS 문의 분석 에이전트 흐름

정책 등록이 완료되면 운영자는 고객 문의와 주문 정보를 분석 API에 전달할 수 있습니다.

### CS 문의 분석 전체 흐름

```text
customer_message 입력
↓
order_context 확인
↓
문의 접수/분류
↓
누락 정보 확인
↓
관련 정책 검색
↓
처리 가능성 판단
↓
고객 답변 초안 작성
↓
답변 안전 검수
↓
운영자 승인 정책 적용
↓
전체 workflow 결과 반환
```

각 단계의 결과는 다음 단계의 입력으로 전달됩니다.

```text
분류 결과
→ 정보 확인과 정책 검색에 사용

누락 정보 + 검색 정책
→ 판단에 사용

판단 결과 + 정책 근거
→ 답변 작성에 사용

답변 초안 + 판단 결과 + 누락 정보 + 정책 근거
→ 안전 검수에 사용
```

## 7. CS 문의 분석 내부 함수/로직 설명

### 7-1. 문의 접수/분류 로직

고객 문의에서 명확한 업무 키워드를 먼저 찾아 유형을 분류합니다. 클레임과 불량처럼 위험도가 높은 유형을 우선 확인하고, 취소와 배송이 함께 언급된 경우에는 고객의 요청인 취소를 우선합니다.

분류 가능한 유형은 다음과 같습니다.

```text
refund
→ 환불

exchange
→ 교환

shipping
→ 배송

cancellation
→ 취소

product_question
→ 상품 문의

defective_item
→ 불량/파손

complaint
→ 불만/클레임

other
→ 기타
```

규칙만으로 유형을 정하기 어려운 문의는 LLM에 전달하며 다음 형식의 JSON 객체를 반환받습니다.

```json
{
  "category": "exchange",
  "category_label": "교환",
  "inquiry_summary": "사이즈가 맞지 않아 교환을 요청한 문의",
  "urgency": "low"
}
```

LLM 호출이나 JSON 파싱에 실패하면 해당 문의를 기타 유형으로 안전하게 처리합니다.

#### 발표에서 설명할 포인트

고객이 작성한 비정형 문장을 바로 답변 생성에 사용하지 않고, 먼저 표준 카테고리와 문의 요약으로 정리합니다. 뒤 단계가 일관된 기준으로 동작하기 위한 첫 번째 정규화 단계입니다.

### 7-2. 정보 확인 로직

문의 유형에 따라 판단에 필요한 정보가 있는지 검사합니다.

```text
환불/교환
→ 수령일
→ 사용 여부
→ 택 제거 여부

불량/파손
→ 불량 또는 파손 상세 내용
→ 주문번호 또는 상품명

배송
→ 주문번호
→ 배송 상태 또는 운송장 번호

취소
→ 주문번호
→ 출고 또는 배송 시작 여부
```

정보는 고객 메시지와 `order_context`에서 함께 찾습니다. 주문번호가 `order_id`, `orderId`, `주문번호` 중 어떤 이름으로 들어오더라도 인식할 수 있도록 필드 별칭을 사용합니다.

사용 여부와 택 제거 여부는 `true`, `false` 같은 값뿐 아니라 `미사용`, `택 그대로`, `택 제거`, `착용했음` 같은 자연어도 규칙으로 해석합니다.

#### 발표에서 설명할 포인트

정보가 부족한데도 AI가 임의로 결론을 내리지 않도록 먼저 누락 정보를 확인합니다. 결과에는 고객에게 추가로 물어봐야 할 항목이 명확한 목록으로 포함됩니다.

### 7-3. 정책 검색 로직

분류된 문의 유형, 고객 메시지, 주문 정보를 합쳐 검색 문장을 만들고 Query Embedding으로 변환합니다.

```text
문의 유형
+ 고객 문의
+ 주문/상품 정보
↓
Query Embedding
↓
ChromaDB 정책 검색
↓
관련 정책 상위 결과
```

정책 컬렉션이 비어 있으면 빈 검색 결과를 반환합니다. ChromaDB 연결 또는 임베딩 호출 중 오류가 발생하면 정책이 없는 것으로 숨기지 않고 정책 검색 오류를 발생시킵니다.

#### 발표에서 설명할 포인트

정책이 실제로 없는 경우와 검색 시스템이 실패한 경우를 구분합니다. 검색 장애를 빈 결과로 처리하면 판단 Agent가 잘못된 결론을 낼 수 있기 때문입니다.

### 7-4. 처리 가능성 판단 로직

판단 Agent는 고객 상황, 누락 정보, 검색된 정책을 비교해 다음 세 단계 중 하나를 반환합니다.

```text
likely_possible
→ 가능성 높음

needs_confirmation
→ 확인 필요

unlikely
→ 가능성 낮음
```

판단 시 적용하는 기본 원칙은 다음과 같습니다.

```text
관련 정책이 없음
→ 확인 필요

필수 정보가 누락됨
→ 확인 필요

사용 또는 택 제거 정황이 있음
→ 교환/환불 가능성 낮음

7일 이내 + 미사용 + 택 유지 조건 충족
→ 가능성 높음

정책의 7일 조건을 초과함
→ 가능성 낮음

배송 문의
→ 실제 배송 상태 확인 필요
```

수령일은 ISO 날짜, `YYYY.MM.DD`, `YYYY년 M월 D일`, 오늘, 어제, 그저께, `N일 전` 등의 표현에서 계산할 수 있습니다.

판단 결과는 위 조건을 사용하는 규칙 기반 로직으로 결정합니다. LLM이 `likely_possible`, `needs_confirmation`, `unlikely` 값을 임의로 변경하지 않습니다.

#### 발표에서 설명할 포인트

처리 결과를 단순한 가능 또는 불가능으로 확정하지 않습니다. 정책 근거와 정보 충족 여부에 따라 가능성 높음, 확인 필요, 가능성 낮음으로 구분해 운영자가 최종 판단할 여지를 남깁니다.

### 7-5. 고객 답변 작성 로직

답변 작성 Agent는 다음 정보를 함께 사용합니다.

```text
문의 유형
+ 고객 문의
+ 주문/상품 정보
+ 누락 정보
+ 검색된 정책
+ 판단 결과
```

답변 작성 규칙은 다음과 같습니다.

```text
친절하고 간결한 한국어 사용
정책 근거를 자연스럽게 포함
무조건·100%·확정·보장 표현 금지
정보가 부족하면 필요한 항목 요청
예외와 최종 처리는 운영자 확인 여지를 남김
```

답변은 첫 번째 정책의 안전한 발췌문, 누락 정보, 판단 결과와 판단 이유를 템플릿에 조합해 작성합니다. 위험 표현이 포함된 정책 문장은 답변 근거 요약에서 제외합니다.

#### 발표에서 설명할 포인트

고객 문의만 보고 바로 답변하지 않습니다. 앞 단계에서 정리한 문의 유형, 정책 근거, 판단 결과를 모두 사용해 답변을 작성합니다.

### 7-6. 안전 검수 로직

작성된 고객 답변을 바로 반환하지 않고 규칙 기반 안전 검수를 수행합니다.

규칙 기반 검수 항목은 다음과 같습니다.

```text
무조건
100%
확정
보장
반드시 가능
절대
법적으로
```

추가로 다음 상황을 위험 요소로 기록합니다.

```text
판단에 필요한 정보가 누락됨
관련 정책 근거가 없음
판단 결과가 가능성 높음이 아님
분쟁 또는 정책 예외 가능성이 있음
```

검수 결과는 다음 형식으로 반환됩니다.

```json
{
  "requires_operator_approval": true,
  "risk_level": "medium",
  "issues": ["판단에 필요한 정보가 일부 누락되어 있습니다."],
  "approval_reason": "운영자 확인이 필요한 CS 응대입니다."
}
```

#### 발표에서 설명할 포인트

템플릿으로 작성된 답변도 그대로 신뢰하지 않습니다. 확답 표현과 정책 근거를 한 번 더 검사하고, 위험한 답변은 운영자 확인 대상으로 전환합니다.

### 7-7. 운영자 승인 로직

안전 검수 결과에 운영자 승인 정책을 추가로 적용합니다.

다음 문의는 운영자 승인이 필요하도록 설정됩니다.

```text
클레임 문의
불량/파손 문의
필수 정보가 누락된 문의
정책 근거가 없는 문의
처리 가능성이 높지 않은 문의
안전 검수에서 위험 요소가 발견된 문의
```

운영자 승인 단계는 별도의 LLM이 실제로 승인하는 단계가 아닙니다. 백엔드가 승인 필요 여부와 사유를 계산해 반환하고, 실제 답변 수정과 승인은 프론트의 운영자가 수행합니다.

#### 발표에서 설명할 포인트

AI가 모든 고객 답변을 자동 확정하는 구조가 아닙니다. 위험하거나 불명확한 문의를 사람이 검토할 수 있도록 명시적으로 구분하는 Human-in-the-loop 단계입니다.

## 8. API 동작 흐름

### 8-1. 정책 등록 API

```http
POST /api/cs/policies/ingest
```

```text
정책 documents 입력
↓
정책 청크 생성
↓
Passage Embedding 호출
↓
ChromaDB 저장
↓
저장 문서 수·청크 수 반환
```

### 8-2. 정책 검색 확인 API

```http
POST /api/cs/policies/search
```

```text
query·category·order_context 입력
↓
Query Embedding 호출
↓
ChromaDB 검색
↓
관련 정책 matches 반환
```

### 8-3. CS 문의 분석 API

```http
POST /api/cs/analyze
```

요청 예시는 다음과 같습니다.

```json
{
  "customer_message": "어제 받은 상품의 사이즈를 교환하고 싶어요. 사용하지 않았고 택도 그대로입니다.",
  "order_context": {
    "order_id": "ORDER-1001",
    "received_days_ago": 1,
    "used": false,
    "tag_removed": false
  }
}
```

응답에는 다음 정보가 포함됩니다.

```text
inquiry_summary
→ 고객 문의 요약

category / category_label
→ 문의 유형

missing_info
→ 추가로 필요한 정보

matched_policies
→ 검색된 정책 근거

decision / decision_label / decision_reason
→ 처리 가능성 판단과 이유

draft_reply
→ 고객에게 보낼 답변 초안

safety_review
→ 위험도와 운영자 승인 필요 여부

workflow
→ 일곱 단계 에이전트 실행 결과
```

## 9. 발표에서 강조하면 좋은 기술 포인트

### 1. 단순 LLM 호출이 아닌 다단계 AI 파이프라인

```text
문의 분류
→ 누락 정보 확인
→ 정책 검색
→ 처리 가능성 판단
→ 답변 작성
→ 안전 검수
→ 운영자 승인 확인
```

한 번의 프롬프트로 답변 전체를 생성하지 않고 단계를 나누어 각 결과를 다음 단계의 근거로 사용합니다.

### 2. 정책 근거를 사용하는 RAG 구조

LLM이 일반적인 쇼핑몰 정책을 추측하지 않도록 운영자가 등록한 실제 정책을 ChromaDB에서 먼저 검색합니다.

### 3. Query와 Passage Embedding의 역할 분리

저장할 정책 문서는 Passage Embedding, 검색할 고객 문의는 Query Embedding을 사용합니다.

### 4. 고객 문의와 정책 데이터의 분리

ChromaDB에는 정책만 저장합니다. 고객 문의 벡터는 검색에만 사용하고 정책 컬렉션에 저장하지 않습니다.

### 5. LLM과 규칙 기반 로직의 역할 분리

명확한 문의 분류는 키워드 규칙으로 처리하고 모호한 분류에만 LLM을 제한적으로 사용합니다. 누락 정보 검사, 날짜 계산, 처리 판단, 답변 템플릿, 위험 문구 검사, 승인 정책은 명시적인 규칙으로 처리합니다.

### 6. 정책 기반의 결정론적 결과

같은 정책과 같은 고객 정보에는 같은 판단과 답변 구조가 나오도록 구성했습니다. LLM이 정책 판단이나 고객 답변의 필수 사실을 새로 만들지 않습니다.

### 7. 운영자 승인 단계

정책 근거가 없거나 분쟁 위험이 있는 답변은 자동 확정하지 않고 운영자가 확인하도록 분류합니다.

## 10. 발표용 핵심 문장

이 CS 에이전트의 핵심은 LLM에게 고객 답변을 한 번에 작성시키는 것이 아닙니다.

운영자가 고객 문의와 주문 정보를 입력하면 먼저 문의 유형을 분류하고, 처리 판단에 필요한 수령일, 사용 여부, 택 제거 여부 등의 정보가 빠졌는지 확인합니다.

그다음 고객 문의를 Query Embedding으로 변환해 ChromaDB에 저장된 쇼핑몰 정책과 비교하고, 검색된 정책과 고객 상황을 바탕으로 처리 가능성을 가능성 높음, 확인 필요, 가능성 낮음으로 판단합니다.

판단 결과와 정책 근거를 이용해 고객 답변 초안을 작성한 뒤, 확답 위험과 분쟁 가능성을 다시 검수합니다. 위험하거나 불명확한 문의는 운영자 승인 대상으로 분류합니다.

즉, 이 기능은 단순한 고객 답변 생성기가 아니라, **문의 분류부터 정책 검색, 판단, 답변 작성, 안전 검수, 운영자 확인까지 연결한 정책 기반 CS 에이전트 워크플로우**입니다.
