# SellerBuddy 프로젝트 목적과 개발 방향

## 1. 프로젝트 목적

SellerBuddy는 실제 상용 서비스 출시보다 **AI 에이전트를 설계하고 프론트엔드와 백엔드를 연결해 동작시키는 경험**을 목표로 하는 프로젝트다.

개인 쇼핑몰 운영자의 마케팅 및 고객응대 업무를 AI 에이전트가 보조하는 과정을 하나의 실행 가능한 데모로 보여준다. 최종 결과물은 프론트엔드와 백엔드 소스를 함께 제공하고, Docker Compose로 쉽게 실행할 수 있는 단일 저장소를 지향한다.

## 2. 현재 제공하려는 기능

- 상품 정보를 바탕으로 Threads 마케팅 문구 생성
- 필요 시 Threads 게시 API 연동
- 쇼핑몰 CS 정책 등록 및 검색
- 고객 문의 분류와 정책 기반 답변 초안 생성
- 운영자가 결과를 확인할 수 있는 프론트엔드

## 3. 고객 문의 입력에 대한 현재 가정

현재 단계에서는 고객 쇼핑몰이나 외부 문의 시스템과 직접 연동하지 않는다.

- 고객사로부터 문의 데이터가 이미 전달되었다고 가정한다.
- 프론트엔드에서 문의 내역을 CSV 파일로 입력받는다.
- CSV로 받은 문의를 백엔드 CS 에이전트가 분석하고 답변 초안을 만든다.
- 웹훅, 고객사 DB 연결, 주문 시스템 연동, 답변 자동 발송은 구현 범위에서 제외한다.

CSV 입력은 실제 고객 시스템 연동을 대체하는 데모용 경계이다. 따라서 현 단계의 핵심은 문의 수집 방식이 아니라, 입력된 문의를 AI 에이전트가 어떻게 처리하는지 보여주는 것이다.

## 4. 도커화 목표

프론트엔드와 백엔드를 하나의 저장소에 포함하고 다음 명령으로 전체 데모를 실행할 수 있게 한다.

```sh
cp .env.example .env
docker compose up --build
```

목표 실행 구조는 다음과 같다.

```text
브라우저
  -> frontend
      -> CSV 문의 입력 및 결과 화면
      -> /api 요청
  -> backend (FastAPI)
      -> AI 에이전트
      -> SQLite 및 ChromaDB
      -> 외부 AI·검색·Threads API
```

기본 구성에서는 프론트엔드가 사용자의 진입점이 되며, 프론트엔드의 `/api` 요청을 Docker 내부의 백엔드로 전달한다. SQLite와 ChromaDB 데이터는 Docker named volume에 저장한다. API 키와 설정값은 이미지에 넣지 않고 `.env`로 주입한다.

## 5. 저장소 방향

발표·학습용 배포 편의성을 위해 프론트엔드와 백엔드를 하나의 저장소로 통합한다.

```text
SellerBuddy/
├── frontend/           # 통합된 프론트엔드 소스와 Dockerfile
├── app/                # FastAPI 백엔드 소스
├── Dockerfile          # 백엔드 이미지
├── Caddyfile           # 프론트와 API 요청 라우팅
├── compose.yaml        # proxy, frontend, api 통합 실행
├── .env.example
├── README.md
└── PROJECT_DIRECTION.md
```

소스 납품을 가정하므로 사용자는 저장소를 받은 뒤 로컬에서 이미지를 빌드한다. 별도의 이미지 레지스트리 배포는 필수 목표가 아니다.

## 6. 현재 범위에서 제외하는 항목

다음 항목은 AI 에이전트 데모 완성에 필요하지 않으므로 우선 구현하지 않는다.

- 고객 쇼핑몰, CRM, 주문 DB와의 실제 연동
- 고객 문의의 실시간 수집 또는 웹훅 처리
- 생성된 CS 답변의 자동 발송
- 로그인, 세부 권한 및 멀티테넌시
- 결제, 라이선스 서버 및 사용량 과금
- Kubernetes, 무중단 배포 및 대규모 확장
- 완전한 운영 모니터링 및 자동 백업 체계

단, 실제 API 키가 Git이나 이미지에 포함되지 않도록 하는 기본적인 비밀값 분리는 유지한다.

## 7. MVP 완료 기준

다음 조건을 만족하면 현재 프로젝트 목적에 맞는 도커화가 완료된 것으로 본다.

1. 프론트엔드와 백엔드가 하나의 저장소에 있다.
2. 새 환경에서 `docker compose up --build`로 두 서비스가 실행된다.
3. 브라우저에서 프론트엔드에 접속할 수 있다.
4. 프론트엔드가 동일 origin의 `/api` 경로로 백엔드를 호출한다.
5. CSV 문의 입력과 CS 분석 결과 확인이 정상 동작한다.
6. 핵심 마케팅 기능도 프론트엔드에서 호출할 수 있다.
7. 컨테이너를 재생성해도 SQLite와 ChromaDB 데이터가 유지된다.
8. README만 보고 환경변수 설정, 실행, 종료 및 오류 확인이 가능하다.

## 8. 다음 작업 우선순위

가장 먼저 **프론트엔드 소스를 이 저장소의 `frontend/` 아래에 준비하고 실행 방식을 분석**한다.

확인할 항목은 다음과 같다.

1. 프론트엔드 프레임워크와 Node 버전
2. npm, yarn, pnpm 중 사용하는 패키지 매니저와 lockfile
3. 개발 및 production build 명령
4. production build 결과 디렉터리
5. 현재 백엔드 API 주소 설정 방식
6. CSV 업로드 후 호출하는 API 경로와 요청 형식

분석이 끝나면 아래 순서로 진행한다.

1. 프론트엔드 production build가 로컬에서 성공하도록 정리
2. 프론트엔드용 multi-stage Dockerfile 작성
3. 프론트엔드 API 주소를 상대 경로 `/api`로 통일
4. 정적 서버에서 `/api`를 `api:8000`으로 프록시
5. `compose.yaml`에 `frontend` 서비스 추가
6. 통합 실행 후 CSV 기반 CS 흐름 smoke test
7. 루트 README에 설치와 실행 방법 정리

향후 작업을 시작할 때는 이 문서를 기준으로 범위를 판단하며, 실제 고객사 시스템 연동이나 상용 운영 기능으로 범위를 확장하지 않는다.
